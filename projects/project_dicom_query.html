DICOM
Extract Data from PACS

1 Abstract

Scrap DICOM from a PACS is a hard process. The lack of documentation and examples make this kind of data mining difficult. The Goal of this work is to show you the basic idea behide DICOM and how to get these data. Some pitfalls will be descibed.

2 Motivation

Digital Imaging and COmmunication in Medicine (DICOM) is the most used model to store and query medical data. Understanding DICOM will open the door to the medical data world.

DICOM model

DICOM model is based on nested data. You can see a "first layer" Data Element (patient_name). A DICOM object can be nested into a Sequence in order to save multiple Sequence of DICOM Object (Like a multi sequence IRM with multiple images per Object). Any Data can be saved using private tag (like a movie in MP4).

DICOM data elements

DICOM Data Element as encoded as a dictionary, Key: address of the element, Value: Value group.

Value group is composed of a value representation (What is this data: here Person Name), Value Lenght and Value (the data itself)

As a result, DICOM format a very adaptable but harder to acces. As Example, the OS is not natively compatible, you need special viewer).

DICOM basic exchange

Communication is embeded in DICOM format (the COM of DICOM). The request system work as the TCP/IP protocole. The client ask the server for a service.

DICOM request message model

The communication configuration is embeded into the DICOM Associate Request Message. The Presentation Context contain the transfert syntax wich is the type of data to transfert (MRI, US, Scaner...).

As an example: a radiologist viewer ask the printer for a US print. The printer validate the transfert if it can print US image.

The Service Class User SCU and Service Class Provider set the role of each partner. With the same example, the SCU is the viewer (ask for a service) and the printer is the SCP (provide a print service).

DICOM exchange protocole

After the association establishment, the data block are sends. An Acknowledgement close the At the end of the transaction.

Note: end of transaction process is embbeded in PyNetDicom. If you build you own comunication process don't forget to close the service without response (after a while).

PACS query process

What I've learned for real world PACS:

1/ Querying a PACS with the StudyInstanceUID (study identifer) from the RIS fail most of the time. It's usually due to PACS/RIS segmentation and server migrations.

2/ PACS query system with DICOM query is limited. It's not possible to select an exam type (arthro scaner)

3/ Query the PACS with Accession Number can fail (testing result with around 20% of empty DICOM)

The process I used:

1/ Query the RIS and apply the filter you're interessed in (usually SQL) and get back the Accession Number (examen identifier). With you list of AN. (Note: an AN is a sensible Data)

2/ Query the PACS with the AN to get all StudyInstanceUID linked (C-FIND).

3/ Query the PACS with the StudyInstanceUID (C-FIND).

4/ The AN can be linked to study you don't need. Example: an AN of a mamography can be compose of the first mamography images, plus an US imges. Filter DICOM you don't need.

C-MOVE process

Most of the time, C-GET won't work. C-FIND ask the PACS to send back the DICOM you looking for. After the association establishment, open a listen service (STORE-SCP) to receive the DICOM from the PACS.
